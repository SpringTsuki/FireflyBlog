
### 20. tail 是什么，它常用的使用场景有哪些？语法如何？

```bash
tail 是一个非常实用的命令行工具，主要用于查看文件末尾的内容。在日常运维和开发中，它尤其适合处理日志文件。常用参数如下：

tail -n 50 access.log
```

-n <行数>        显示文件末尾指定的行数（默认是10行）。

`tail -f /var/log/syslog`

-f        实时跟踪文件新增内容。通常在排查问题时保持这个命令运行，观察新日志。


`tail -F messages`

-F        功能与 -f 类似，但如果文件被重命名或轮转（比如日志切割），-F 仍然能继续跟踪新的同名文件。


`tail -q file1.txt file2.txt`

-q        如果同时查看多个文件，可以隐藏文件名头，让输出更干净。


`tail -v access.log error.log`

-v        无论查看几个文件，都强制显示文件名头。


`tail -c 1000 data.log`

-c <字节数>        显示文件末尾指定的字节数，通常用得较少。

### 21. iptables怎么设置只放行某个ip的访问？

记住思路就好，首先要把所有的流量访问全部drop，其次建立一个优先级比它高的规则放行指定 tcp ip 的访问。


```bash
# 1. 清空现有规则（谨慎操作，建议在测试环境或通过带外管理执行）
iptables -F

# 2. 设置默认策略：拒绝所有入站流量
iptables -P INPUT DROP

# 3. 放行回环接口（127.0.0.1），避免本地服务出问题
iptables -A INPUT -i lo -j ACCEPT

# 4. 放行已建立连接的相关流量（重要！否则已建立的连接会被中断）
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# 5. 放行指定 IP 的访问（以 22 端口为例）
iptables -A INPUT -p tcp --dport 22 -s 192.168.1.100 -j ACCEPT

# 6. （可选）放行 ICMP，允许 ping
iptables -A INPUT -p icmp -j ACCEPT
```

### 25. 监控 CPU 的 top 指令详解

### 1. Linux 平均负载（load average）是什么？三个数值高低分别代表什么情况？

load average 反映的是系统在特定时间段内的平均活跃进程数，包括正在运行和等待CPU的进程。对于多核系统，需要将数值除以核心数来理解。例如，4核系统负载为4.0，相当于每个核心跑满。我通常会对比1分钟、5分钟和15分钟三个数值，来判断负载趋势。如果负载持续高于核心数，我会用 top 或 ps 定位具体进程，判断是CPU瓶颈还是IO瓶颈，再针对性处理。

### 4. %us、%sy、%id、%wa 分别代表什么含义？

%us（用户空间 CPU 占用率）：好比 厨师实际在做菜的时间。这是应用程序（Python、Nginx、MySQL）真正在执行计算指令的时间。数值高通常意味着你的应用逻辑本身很复杂，是计算密集型任务，CPU 正在卖力工作，这是硬件瓶颈的信号。

%sy（内核空间 CPU 占用率）：好比 厨师在整理厨具、看菜单、调度任务的时间。这是操作系统内核在管理硬件资源、处理系统调用的时间。数值高可能意味着程序在进行大量的文件读写、网络通信或进程切换。如果这个值持续偏高，通常说明系统架构存在优化空间。

%id（空闲 CPU）：好比 厨师在等菜、闲着发呆的时间。空闲时间越多越好，说明服务器有余力。

%wa（IO 等待 CPU 占用率）：好比 厨师在等配菜员切菜的时间。CPU 已经发出了读写磁盘的指令，但迟迟等不到数据返回，此时 CPU 本身是空闲的，但处于等待状态。高 %wa 通常意味着磁盘速度太慢，或者内存不足导致频繁的硬盘交换。


记忆方法：us（正在占用）、system（内核调度）、id（空闲时段）、wating（IO等待）

### 5. 进程状态 R、S、D、Z、T 分别代表什么？

Running：正在运行或等待CPU调度的进程。进程要么正在CPU上执行，要么在运行队列中等待。

Sleeping：可中断睡眠。进程在等待某个事件完成（如用户输入、网络请求、文件读取），可以被信号中断唤醒。这是大多数进程的常见状态。

D - Uninterruptible Sleep：不可中断睡眠。进程在等待硬件响应（如磁盘I/O），无法被信号中断。这种状态常见于磁盘读写操作，如果大量出现D状态进程，通常意味着I/O存在瓶颈。

Zombie：僵尸进程：进程已经终止，但父进程尚未调用wait()来获取其退出状态。僵尸进程不消耗资源，但会占用进程表条目。如果数量过多，需要检查父进程是否正常回收。

T - Stopped：停止或跟踪状态。进程被暂停（如收到SIGSTOP或SIGTSTP信号），或正在被调试器追踪（如gdb）。

### 26. 什么是软链接，什么是硬链接？区别如何？

硬链接本质上是同一个文件的不同名字，删除原文件不影响链接，但不能跨分区也不能指向目录。软链接是一个独立的文件，指向目标路径，可以跨分区也可以指向目录，但原文件被删了就会失效。我一般用软链接来做快捷方式或版本切换，硬链接通常用于备份或引用同一份数据。

软链接

程序多版本切换（比如 python3、python 软链接切换版本）

目录路径简化、挂载路径迁移兼容

方便部署配置、日志路径统一管理

硬链接

简单本地文件备份，删一个入口数据仍保留

防止误删重要文件，多入口保护

### 27. 简述软 RAID 和硬 RAID 的区别，各自优缺点？

简单说，软RAID靠CPU和操作系统算，硬RAID靠专用的RAID卡算。

⚙️ 软RAID (Software RAID)：是通过操作系统（如Windows、Linux）和CPU来管理磁盘阵列的。

优点：

成本低：无需购买独立的RAID卡，使用主板上的SATA接口即可。

灵活性高：与操作系统结合紧密，管理和配置相对简单。可以比较方便地将磁盘阵列迁移到另一台兼容的系统上。

无厂商绑定：不依赖特定品牌的硬件，减少了被特定厂商锁定的风险。

缺点：

消耗CPU资源：RAID的计算任务会占用服务器的CPU和内存，在高负载下可能影响其他业务性能。

依赖操作系统：如果操作系统崩溃或配置错误，RAID的管理和数据恢复会变得复杂。

系统盘限制：在一些操作系统中，软RAID可能无法保护系统启动盘。

适用场景：适合预算有限、CPU算力充裕的环境，如个人电脑、实验室、中小型企业（SMB）服务器，以及对灵活性要求较高的场景。

🖥️ 硬RAID (Hardware RAID)：通过独立的RAID控制器（RAID卡）来管理磁盘，这个控制器拥有自己的处理器、内存和缓存。

优点：

性能好：RAID计算由专用处理器完成，不占用主机CPU资源，性能表现更稳定。高端RAID卡的缓存能进一步提升性能。

更可靠：独立于操作系统运行，兼容性更好。部分RAID卡带有电池或电容，能在断电时保护缓存中的数据。

支持热插拔：更换故障硬盘更方便。

缺点：

成本高：需要额外购买独立的RAID卡，增加了硬件成本。

配置复杂：设置和管理通常需要专业技能。

存在厂商锁定：如果RAID卡损坏，可能需要更换同型号的卡才能恢复数据。

适用场景：适合对性能和数据安全要求严苛的生产环境，如大型企业服务器、数据库服务器等。

✨ 总结

软RAID：是一个性价比高、灵活的软件方案。硬RAID：是一个性能强劲、可靠的硬件方案。

### 28. RAID0、RAID1、RAID5、RAID10 各自原理、磁盘数量要求、优缺点、适用场景？

RAID 0（条带化）

原理：数据被分成块，交替写入多块磁盘，并行读写。

磁盘数量：至少 2 块（可更多）。

优点：读写性能最高，容量利用率 100%。

缺点：没有任何冗余，一块磁盘损坏，所有数据全部丢失。

场景：数据丢失无所谓、追求极致速度的场景（如视频编辑缓存、游戏盘）。

RAID 1（镜像）

原理：数据同时写入两块磁盘，互为镜像。写一次同时写两份，读可以从两块盘同时读。

磁盘数量：至少 2 块（通常只能 2 块）。

优点：数据冗余最高，一块盘坏不影响数据，读性能有所提升。

缺点：容量利用率只有 50%（2块盘只能用到1块的容量），写入性能稍有下降。

场景：对数据安全性要求极高、数据量不大的场景（如数据库日志盘、系统盘）。

RAID 5（条带化+奇偶校验）

原理：数据条带化分布，同时插入一份奇偶校验数据。校验数据分布在所有磁盘上，可以容忍一块磁盘损坏。

磁盘数量：至少 3 块（可更多）。

优点：兼顾性能、容量和安全，容量利用率 (n-1)/n，坏一块盘时数据仍可用。

缺点：写入性能较差（每次写需要计算校验数据并写入），重建恢复过程比较耗时。

场景：企业通用存储，适合大多数生产环境（文件服务器、应用服务器）。

RAID 10（RAID 1 + RAID 0）

原理：先做 RAID 1（镜像），再将多个 RAID 1 组合成 RAID 0（条带化）。它集合了 RAID 0 的速度和 RAID 1 的冗余。

磁盘数量：至少 4 块（且必须为偶数）。

优点：读性能极高，写性能优秀，容错能力好（每一组镜像中可坏一块）。

缺点：成本高，容量利用率只有 50%（4块盘实际可用容量为2块）。

场景：对性能和安全性要求都很高的核心数据库、高并发应用服务器。

### 29. LVM是什么？

LVM 逻辑卷管理器（Logical Volume Manager），是 Linux 下对磁盘资源进行弹性管理的机制，在物理磁盘分区之上抽象一层逻辑层，不再受原有分区大小限制。以下为优点：

可以在线扩容卷空间，不用重装系统、不用迁移数据；

支持后期缩减逻辑卷（谨慎操作）；

多块硬盘整合为一个大存储池，灵活分配空间；

业务数据增长时按需扩容，适配业务长期扩容需求。

### 30. 什么是缓冲区（buffer）和缓存（cache）？free 命令里怎么区分？

Buffer 是磁盘写入时的暂存区，用于合并写操作；Cache 是文件的读缓存，用于加速重复读取。在 free 命令中，它们合在一起显示为 buff/cache，但 available 才是真正可用的内存，因为系统在需要时会自动释放一部分缓存。如果需要查看具体值，可以看 /proc/meminfo。

Buffer（缓冲区）：

存储内容：存储的是元数据，主要是文件系统的数据块、目录信息等。

作用：减少磁盘的随机读写。例如，当你写入数据时，数据先进入Buffer，积累到一定量再一次性写入磁盘。

通俗理解：像是"快递单"上的信息，帮助系统快速找到和整理磁盘上的文件。

Cache（缓存）：

存储内容：存储的是实际数据，是文件的正文内容。

作用：减少磁盘的重复读取。当你多次打开同一个文件时，缓存可以避免反复从硬盘读取。

通俗理解：像是用过的"实物快递"，最近使用过的文件内容被暂存起来，方便下次直接取用。

### 33. 孤儿进程是什么？

孤儿进程是指父进程已经退出，但子进程仍在运行的进程。系统会自动将其父进程设为init（PID 1），由init负责回收其资源，因此孤儿进程本身不会造成资源泄漏。与僵尸进程不同，孤儿进程是活的，不需要特殊处理

### 1. 简述 TCP 和 UDP 的区别，各自适用场景？

文件传输多用 TCP，视频直播、语音通话多用 UDP

TCP 面向连接 通信前三次握手建立连接，传输结束四次挥手断开； 可靠传输：有确认应答、重传机制、流量控制、拥塞控制，数据不丢包、不乱序。 缺点：开销大，速度稍慢。 适用场景： 文件传输、SSH 远程登录、网页 HTTP/HTTPS、数据库连接、FTP、邮件，要求数据完整准确不能出错的业务。

UDP 无连接 收发直接发包，不用建立连接，没有重传、校验纠错机制； 不可靠，可能丢包、乱序；优点：头部小、延迟极低、收发效率高。 适用场景： 直播、语音视频通话、游戏实时数据包、DNS 查询，能容忍少量丢包、优先低延迟的场景。

### 2. TCP 的三次握手与四次挥手的报文单词是什么？

SYN：建立连接同步

ACK：确认收到报文

FIN：请求断开连接

RST：异常强制断连

### 3. TIME_WAIT 是做什么的？

TIME_WAIT 是主动关闭方在发送最后一个 ACK 后进入的状态，持续 2MSL，默认通常是 60 秒。它的作用是确保最后的 ACK 能被对方收到，以及让网络中残留的旧数据包在超时后失效，避免干扰后续的新连接。

### 4. HTTP 和 HTTPS 的区别如何？它们是如何工作的？

HTTP 是一种基于 TCP 的应用层协议，客户端发送请求，服务器返回响应。每个请求报文包含请求方法、路径和头部信息，响应报文包含状态码和头部信息。数据在传输前会经过 TCP 分片、IP 封装、链路层成帧，最终在网络中传输。HTTPS 在 HTTP 和 TCP 之间加入了 TLS/SSL 加密层。


一次完整的 HTTP 请求流程如下：

域名解析：客户端通过 DNS 将域名解析为服务器的 IP 地址。

建立 TCP 连接：客户端与服务器进行三次握手，建立 TCP 连接（默认端口 80 或 443）。

发送 HTTP 请求：客户端发送 HTTP 请求报文，包含：

请求行：方法（GET/POST）、路径、协议版本。

请求头：Host、User-Agent、Accept、Cookie 等。

请求体（可选）：POST、PUT 等请求的数据内容。

服务器处理请求：服务器接收到请求后，解析报文，定位资源，执行业务逻辑（如查询数据库、调用后端服务）。

返回 HTTP 响应：服务器发送响应报文，包含：

状态行：协议版本、状态码（200、404、500 等）、状态描述。

响应头：Content-Type、Content-Length、Set-Cookie 等。

响应体（可选）：请求的资源内容（如 HTML、JSON、图片）。

关闭或复用连接：如果是非持久连接（HTTP/1.0），服务器主动关闭 TCP 连接；如果是持久连接（HTTP/1.1 默认），连接保持打开，以便后续请求复用。

### 5. GET 请求与 POST 请求有什么不同？

GET：设计用于查询、获取数据，理论上不应修改服务器数据，是安全、幂等请求（多次请求结果一致）

POST：设计用于提交数据、新增 / 修改资源，提交表单、上传数据，非幂等，多次提交可能产生多条数据

### 6. Nginx 反向代理和正向代理的区别，反向代理有什么作用？

### 1. 正向代理

代理客户端，隐藏真实访问用户；服务端只看到代理 IP，无法获知原始客户端是谁。

典型场景：VPN、翻墙代理、局域网统一上网代理。

访问链路：客户端 → 正向代理服务器 → 目标网站

### 2. 反向代理

代理后端服务器集群，隐藏后端真实业务节点；客户端只访问代理地址，感知不到后端多台机器。

典型场景：Nginx 反向代理、负载均衡、统一入口、SSL 终结。

访问链路：客户端 → 反向代理 (Nginx) → 后端应用服务器

### 1. CVE-2026-31431 "Copy Fail"漏洞防御

禁用 authencesn、authenc、esp4 和 esp6 这几个内核模块，确实是针对 CVE-2026-31431（Copy Fail）漏洞的一种有效临时缓解方案。这些模块，尤其是 authencesn，与漏洞利用密切相关。

我根据你的思路，并参考了多个官方安全公告，将具体操作整理成了下面这份更完善的步骤指南。


📝 漏洞修复操作步骤（临时缓解方案）

#### 第一步：创建配置文件，禁用模块

首先，需要创建一个配置文件，让系统在启动时就不再加载这些有漏洞的模块。

使用 root 权限创建一个新的配置文件，例如 /etc/modprobe.d/disable-cve-2026-31431.conf：

bash

`sudo vi /etc/modprobe.d/disable-cve-2026-31431.conf`

在文件中写入以下内容，以禁用 authencesn、authenc、esp4 和 esp6 这四个模块：

conf

`install authencesn /bin/falseinstall authenc /bin/falseinstall esp4 /bin/falseinstall esp6 /bin/false`


命令解释：install 模块名 /bin/false 是告诉系统，当有程序试图加载这个模块时，执行 /bin/false 命令（该命令会立即返回失败），从而阻止模块加载。

#### 第二步：卸载当前已加载的模块

配置文件创建后，需要立即卸载当前已经加载到内核中的这些模块。


bash

`sudo rmmod authencesn authenc esp4 esp6 2>/dev/null || true`


命令解释：rmmod 用于从运行中的内核卸载模块。2>/dev/null || true 的作用是忽略可能出现的"模块不存在"等错误信息，确保命令在任何情况下都能顺利执行。

#### 第三步：验证修复是否生效

最后，检查一下模块是否已被成功禁用和卸载。


bash

`lsmod | grep -E "authencesn|authenc|esp4|esp6"`

如果命令没有任何输出，说明相关模块已经成功卸载，并且由于配置文件的设置，系统重启后也不会再加载它们。
